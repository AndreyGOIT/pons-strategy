# Sprint 1 Reflection

## 1. Sprint Focus

Sprint 1 focused on understanding, structuring, and documenting the architecture of the Pons.fi system.

The main objective was not to add new functionality, but to establish a professional foundation for further development and näyttö preparation.

---

## 2. Main Activities

Main activities during Sprint 1 included:

- analyzing the current system structure
- documenting frontend and backend architecture
- documenting database structure
- defining Git workflow principles
- creating ADR documentation
- organizing project documentation structure
- mapping work to näyttö criteria

---

## 3. Architectural Understanding

One of the most important outcomes of Sprint 1 was gaining a clearer understanding of the system architecture.

This included:

- separation between frontend and backend
- API-based communication
- role of the database layer
- routing and middleware structure
- deployment-related configuration

The architecture became easier to explain, maintain, and improve.

---

## 4. Documentation as Engineering Work

A key realization during Sprint 1 was that documentation itself is an important part of professional software development.

Creating structured documentation helped:

- clarify architectural decisions
- identify weaknesses and inconsistencies
- improve maintainability
- prepare for future refactoring

---

## 5. Challenges

Main challenges included:

- translating an organically grown project into a structured architecture
- balancing technical detail with readability
- deciding how much documentation was necessary
- connecting practical implementation with näyttö requirements

---

## 6. Solutions

The following approaches helped address these challenges:

- incremental documentation
- sprint-based structure
- separating implementation and strategy repositories
- focusing on maintainability instead of perfection

---

## 7. Professional Growth

Sprint 1 improved understanding of:

- software architecture
- separation of concerns
- professional Git workflows
- documentation practices
- long-term maintainability

The sprint also highlighted the importance of viewing software as part of a larger operational system rather than only as code.

---

## 8. Impact on Project Direction

Sprint 1 created the foundation for all later improvements.

Without architectural clarity and documentation, later work on error handling, validation, deployment, and security would have been more difficult and less consistent.

---

## 9. Summary

Sprint 1 transformed the project from a functional application into a more structured engineering project.

The sprint established architectural clarity, documentation practices, and a professional development direction for the rest of the project.
